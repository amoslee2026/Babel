### ISSUE-CON-02 [HIGH] — External Clock Frequency: PRD vs ARCH vs Netlist

| Aspect | Value | Source |
|--------|-------|--------|
| ARCH §5.1 | ext_clk = 25 MHz | arch_doc.md |
| Netlist NPU_top | ext_clk_50MHz | synth/netlist |
| MAS §3.7 | ext_clk_i (frequency unspecified) | mas.md |

**Finding**: ARCH specifies 25 MHz external oscillator but netlist uses 50 MHz pin. MAS is ambiguous.

**Recommendation**: Standardize to 25 MHz external oscillator (as ARCH specifies). The PLL multiplies to VCO frequency; 25 MHz is a common crystal frequency. Update MAS §3.7 to note `ext_clk_i` = 25 MHz. The netlist's 50MHz pin name is from a previous generation and should be renamed to 25MHz in RTL re-generation.

**Severity**: HIGH — downstream RTL must use consistent frequency.

### 3.4 Feasibility Issues

### ISSUE-FEAS-01 [HIGH] — M11 and M12 are New RTL; No Reference Implementation

| Aspect | Detail |
|--------|--------|
| Previous RTL | M11 was "RMSNormRoPE" (combined), M12 was "SoftMax" (separate) |
| New MAS | M11 = RMSNorm (standalone), M12 = RoPE + SoftMax (combined) |
| Impact | bba-guru-rtl must create RTL from scratch for both modules |

**Finding**: The module split is architecturally sound (see MAS-ADR-001), but creates development risk. No existing RTL to adapt from (old M11 combined different functions, old M12 was SoftMax-only).

**Recommendation**: Provide detailed algorithmic reference in datapath docs (already done in MAS §3.12-3.13 and datapath/). Accept the risk; flagged for bba-guru-rtl attention.

**Severity**: HIGH — requires new RTL development for two modules.

### ISSUE-FEAS-02 [MEDIUM] — 1024-bit SRAM Read Port in ASAP7

| Aspect | Detail |
|--------|--------|
| Requirement | Bank 0/1: 1024-bit read port |
| ASAP7 | Standard SRAM macros likely limited to 64-256 bit wide |

**Finding**: 1024-bit wide SRAM read port is unusual. May require:
- 4×256-bit macros interleaved → adds 4-way interleaving logic
- Register file implementation → increases area significantly
- Custom SRAM compiler → not available in open-source flow

**Recommendation**: Document fallback plan in MAS: use 4-way interleaved 256-bit SRAM macros with address interleaving. The 4:1 width conversion is acceptable for the systolic array weight load (32 cycles already budgeted for weight loading; interleaving would add at most 4 cycles overhead).

**Severity**: MEDIUM — affects area estimate, not functionality.

## 5. Ambiguity Check

### ISSUE-AMB-01 [MEDIUM] — FP8 E4M3 Special Value Handling Partially Specified

| Aspect | Detail |
|--------|--------|
| Specified | NaN, +/-Inf, subnormals, encoding table (§11.2) |
| Missing | Denormal flush-to-zero policy, NaN quieting rules, signed zero behavior |

**Finding**: MAS §11 covers FP8 format well but misses edge cases:
- Should subnormal inputs be flushed to zero (FTZ) or processed?
- Which NaN encoding is canonical (0x7F or 0xFF)?
- How is signed zero handled in comparisons?

**Recommendation**: Add 2-3 sentences clarifying these policies. Standard ML practice: FTZ for subnormals (performance), canonical NaN = 0x7F.

**Severity**: MEDIUM — RTL coder needs unambiguous specification for FP8 arithmetic.

### ISSUE-AMB-02 [LOW] — SHA256 Placeholders in mas.json

| Aspect | Detail |
|--------|--------|
| mas.json inputs[].sha256 | Placeholder values (all same repeated characters) |
| Required | Real SHA256 hashes for downstream drift detection |

**Recommendation**: Compute real SHA256 hashes of all MAS source files before handoff. The placeholder values match the schema pattern but won't detect real drift.

**Severity**: LOW — does not affect RTL coding; affects drift detection in later pipeline stages.
